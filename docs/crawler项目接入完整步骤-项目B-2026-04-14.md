# Crawler 项目接入 GEO-watcher 完整步骤

本文用于把一个新的爬虫项目接入 GEO-watcher。当前已接入的爬虫项目记为“项目 A”，新的爬虫项目记为“项目 B”。

目标部署形态：

- GEO-watcher 和爬虫项目部署在同一台 Linux 服务器
- 所有服务都使用 Docker Compose 部署
- GEO-watcher 对外仍然只通过 nginx 暴露
- GEO-watcher 后端不直接暴露宿主机端口
- crawler 与 GEO-watcher backend 通过 Docker external network 直连
- 项目 B 使用新的 Docker external network，不复用项目 A 的 `crawler-shared`

## 1. 当前项目 A 的现状

项目 A 当前方案是：

- 共享网络：`crawler-shared`
- GEO-watcher backend 在该网络上的别名：`geo-watcher-backend`
- 项目 A crawler API 在该网络上的别名：`geo-watch-crawler`
- GEO-watcher 调项目 A 的地址：

```env
CRAWLER_BASE_URL=http://geo-watch-crawler:8000
```

- 项目 A 回调 GEO-watcher 的地址：

```env
CRAWLER_CALLBACK_URL=http://geo-watcher-backend:3005/api/integrations/crawler/uploads
```

项目 A 的做法是正确的，不需要改成走公网域名，也不需要改成 `127.0.0.1`。

## 2. 项目 B 的推荐命名

为避免和项目 A 混用，项目 B 单独使用一套网络和别名。

推荐值：

```env
PROJECT_B_SHARED_NETWORK=project-b-crawler-shared
PROJECT_B_CRAWLER_ALIAS=project-b-crawler
PROJECT_B_BACKEND_ALIAS=geo-watcher-backend
```

说明：

- `project-b-crawler-shared` 是项目 B 专用 external Docker network
- `project-b-crawler` 是项目 B crawler API 服务在共享网络里的 hostname
- `geo-watcher-backend` 是 GEO-watcher backend 在项目 B 共享网络里的 hostname
- 同一个 alias 可以出现在不同 Docker network 中，因此项目 A 和项目 B 都可以使用 `geo-watcher-backend` 作为回调目标

## 3. 服务器先创建项目 B 的共享网络

在服务器执行：

```bash
docker network create project-b-crawler-shared
```

如果已经存在，会报已存在错误，可以忽略。

验证：

```bash
docker network ls | grep project-b-crawler-shared
```

## 4. GEO-watcher 需要做的改动

### 4.1 修改 `docker-compose.prod.yml`

GEO-watcher 的 `backend` 服务需要额外加入项目 B 的 external network。

示例：

```yaml
services:
  backend:
    networks:
      geo-watcher: {}
      crawler-shared:
        aliases:
          - ${CRAWLER_BACKEND_ALIAS:-geo-watcher-backend}
      project-b-crawler-shared:
        aliases:
          - ${PROJECT_B_BACKEND_ALIAS:-geo-watcher-backend}

networks:
  geo-watcher:
    driver: bridge

  crawler-shared:
    external: true
    name: ${CRAWLER_SHARED_NETWORK:-crawler-shared}

  project-b-crawler-shared:
    external: true
    name: ${PROJECT_B_SHARED_NETWORK:-project-b-crawler-shared}
```

注意：

- 不要把 nginx/frontend/admin-frontend 挂到项目 B 的共享网络
- 只需要 GEO-watcher 的 `backend` 加入项目 B 共享网络
- 这样项目 B 只能直连 backend callback，不会接触前端/nginx 内部网络

### 4.2 修改 GEO-watcher 的 `backend/.env`

保留项目 A 的现有配置：

```env
CRAWLER_SHARED_NETWORK=crawler-shared
CRAWLER_BACKEND_ALIAS=geo-watcher-backend
CRAWLER_BASE_URL=http://geo-watch-crawler:8000
CRAWLER_CALLBACK_URL=http://geo-watcher-backend:3005/api/integrations/crawler/uploads
CRAWLER_CALLBACK_API_KEY=<双方约定的同一串 key>
```

为项目 B 增加网络变量：

```env
PROJECT_B_SHARED_NETWORK=project-b-crawler-shared
PROJECT_B_BACKEND_ALIAS=geo-watcher-backend
```

### 4.3 配置项目 B 的平台 endpoint

GEO-watcher 当前只有一个全局 `CRAWLER_BASE_URL`，它适合默认项目 A。

如果项目 B 也要接入，项目 B 对应平台必须在 `CRAWLER_PLATFORM_ENDPOINTS` 里使用完整 URL，不能依赖全局 `CRAWLER_BASE_URL`。

示例：

```env
CRAWLER_PLATFORM_ENDPOINTS={
  "deepseek":"/api/v1/tasks/deepseek",
  "doubao":"/api/v1/tasks/doubao-stand",
  "yuanbao":"/api/v1/tasks/yuanbao",
  "doubao-ark":"/api/v1/tasks/doubao-ark",
  "doubao-stand":"/api/v1/tasks/doubao-stand",
  "project-b-platform":"http://project-b-crawler:8000/api/v1/tasks/project-b-platform"
}
```

实际 `.env` 里要写成一行：

```env
CRAWLER_PLATFORM_ENDPOINTS={"deepseek":"/api/v1/tasks/deepseek","doubao":"/api/v1/tasks/doubao-stand","yuanbao":"/api/v1/tasks/yuanbao","doubao-ark":"/api/v1/tasks/doubao-ark","doubao-stand":"/api/v1/tasks/doubao-stand","project-b-platform":"http://project-b-crawler:8000/api/v1/tasks/project-b-platform"}
```

说明：

- key 必须等于 GEO-watcher 数据库里的 `platform_id`
- value 如果是相对路径，会拼到 `CRAWLER_BASE_URL`，即项目 A
- value 如果是完整 URL，会直接请求该 URL，适合项目 B

### 4.4 重建 GEO-watcher

```bash
cd ~/watcher-geo
bash start.sh restart
```

如果涉及数据库结构更新，`start.sh` 已经会自动执行：

- `alembic upgrade head`
- 失败后 fallback 到 `python scripts/repair_schema.py --stamp-head`
- `python -m app.initial_data`

如果需要手动补 schema：

```bash
docker compose -f docker-compose.prod.yml exec backend python scripts/repair_schema.py --stamp-head
```

## 5. 项目 B 需要做的改动

### 5.1 修改项目 B 的 compose

项目 B 的 crawler API 服务需要加入项目 B external network，并设置固定 alias。

示例：

```yaml
services:
  api:
    networks:
      project-b-internal: {}
      project-b-crawler-shared:
        aliases:
          - ${PROJECT_B_CRAWLER_ALIAS:-project-b-crawler}

networks:
  project-b-internal:
    driver: bridge

  project-b-crawler-shared:
    external: true
    name: ${PROJECT_B_SHARED_NETWORK:-project-b-crawler-shared}
```

注意：

- `api` 是示例服务名，按项目 B 实际服务名替换
- 项目 B 内部网络保留自己的即可
- 项目 B 不要加入项目 A 的 `crawler-shared`
- 项目 B 只需要加入 `project-b-crawler-shared`

### 5.2 修改项目 B 的 `.env`

项目 B 至少需要：

```env
PROJECT_B_SHARED_NETWORK=project-b-crawler-shared
PROJECT_B_CRAWLER_ALIAS=project-b-crawler
CRAWLER_CALLBACK_URL=http://geo-watcher-backend:3005/api/integrations/crawler/uploads
CRAWLER_CALLBACK_API_KEY=<双方约定的同一串 key>
SCRAPER_API_KEY=<GEO-watcher 调用项目 B 的 x-api-key>
```

说明：

- `CRAWLER_CALLBACK_URL` 指向 GEO-watcher backend
- `geo-watcher-backend` 是 GEO-watcher backend 在项目 B 共享网络里的 alias
- `CRAWLER_CALLBACK_API_KEY` 必须和 GEO-watcher `backend/.env` 里的 `CRAWLER_CALLBACK_API_KEY` 一致
- `SCRAPER_API_KEY` 或同类变量取决于项目 B 自己的鉴权实现，必须和 GEO-watcher 请求项目 B 时使用的 `CRAWLER_API_KEY` 对得上

### 5.3 项目 B 必须提供任务创建接口

GEO-watcher 会向项目 B 发 `POST` 请求。

请求头：

```http
Content-Type: application/json
x-api-key: <GEO-watcher 的 CRAWLER_API_KEY>
```

请求体：

```json
{
  "prompts": ["问句1", "问句2"],
  "repeat": 1,
  "save_name": "2026-04-14_project-b-platform_user_product_keyword_07_r1",
  "env": {
    "task_id": "GEO-watcher task id",
    "monitor_date": "2026-04-14",
    "user_id": "user id",
    "product_id": "product id",
    "keyword_id": "keyword id",
    "platform_id": "project-b-platform",
    "is_negative": "false",
    "run_id": "task_id:2026-04-14:7:1",
    "callback_url": "http://geo-watcher-backend:3005/api/integrations/crawler/uploads",
    "callback_api_key": "<双方约定的同一串 key>"
  }
}
```

项目 B 接到任务后建议立即返回：

```json
{
  "task_id": "project-b-side-task-id",
  "trace_id": "trace-id",
  "type": "project-b-platform",
  "status": "pending",
  "expected_records": 2,
  "output_file": "/app/output/xxx.json"
}
```

GEO-watcher 会把这个响应写入执行日志。

### 5.4 项目 B 完成任务后必须回调 GEO-watcher

回调地址：

```text
POST http://geo-watcher-backend:3005/api/integrations/crawler/uploads
```

请求头：

```http
x-api-key: <CRAWLER_CALLBACK_API_KEY>
```

请求类型：

```text
multipart/form-data
```

表单字段：

```text
run_id
task_id
user_id
platform_id
product_id
keyword_id
monitor_date
files
```

字段来源：

- 全部从 GEO-watcher 派单请求里的 `env` 取
- 不要自己重新生成 `run_id`
- 不要自己改 `monitor_date`
- 不要丢 `task_id`

文件要求：

- `files` 可以是一个或多个 JSON 文件
- 每个文件必须是合法 JSON
- JSON 顶层必须是数组

示例文件：

```json
[
  {
    "query": "用户问句",
    "提及率": 100,
    "前三率": 100,
    "置顶率": 80,
    "负面提及率": 0
  },
  {
    "attitude": 92
  }
]
```

成功返回：

```json
{
  "queued": 1,
  "ids": ["review id"]
}
```

## 6. 连通性验证

### 6.1 验证项目 B crawler 能访问 GEO-watcher backend

进入项目 B crawler 容器：

```bash
docker exec -it <project-b-api-container> sh
```

执行：

```bash
python - <<'PY'
from urllib.request import urlopen
print(urlopen("http://geo-watcher-backend:3005/health", timeout=5).read().decode())
PY
```

期望输出：

```json
{"status":"healthy"}
```

### 6.2 验证 GEO-watcher backend 能访问项目 B crawler

进入 GEO-watcher backend 容器：

```bash
docker exec -it geo-watcher-backend sh
```

执行：

```bash
python - <<'PY'
from urllib.request import urlopen
print(urlopen("http://project-b-crawler:8000/health", timeout=5).read().decode())
PY
```

如果项目 B 没有 `/health`，至少要验证 TCP/HTTP 入口存在，可以改成项目 B 实际健康检查路径。

### 6.3 验证项目 B 共享网络里有两个容器

```bash
docker network inspect project-b-crawler-shared
```

应能看到：

- `geo-watcher-backend`
- 项目 B 的 API 容器

## 7. GEO-watcher 后台验证

1. 在 GEO-watcher 后台创建或编辑平台，确保平台 ID 与 `CRAWLER_PLATFORM_ENDPOINTS` key 一致
2. 在用户/关键词里绑定该平台
3. 在问句管理里给关键词添加问句
4. 创建任务，选择项目 B 对应平台
5. 点击立即执行
6. 在任务日志菜单或任务执行日志里检查：
   - 是否派单成功
   - 是否有 project B 的 crawler task id
   - 是否有 trace id
   - 是否收到 callback
   - 是否进入待审核
   - 是否产生 callback 上传文件 artifact
7. 到审核队列查看项目 B 回传内容
8. 审核通过后检查指标结果是否入库

## 8. 常见错误

### 8.1 `Connection refused`

通常是 GEO-watcher 请求项目 B 的地址不对。

检查：

- `CRAWLER_PLATFORM_ENDPOINTS` 是否用了完整 URL
- URL host 是否是 `project-b-crawler`
- 项目 B 容器是否加入 `project-b-crawler-shared`
- 项目 B 服务是否监听 `0.0.0.0:8000`，不是只监听 `127.0.0.1`

### 8.2 `Name or service not known`

通常是 Docker network 没接对。

检查：

```bash
docker network inspect project-b-crawler-shared
```

确认 GEO-watcher backend 和项目 B API 容器都在里面。

### 8.3 回调 401

说明 `x-api-key` 不一致。

检查：

- GEO-watcher `CRAWLER_CALLBACK_API_KEY`
- 项目 B `CRAWLER_CALLBACK_API_KEY`

两边必须完全一致，不能有尾随空格。

### 8.4 回调 400

常见原因：

- `monitor_date` 不是 `YYYY-MM-DD`
- `user_id/platform_id/product_id/keyword_id` 不存在
- `files` 不是合法 JSON
- JSON 顶层不是数组
- project B 没有原样使用 GEO-watcher 下发的 `env`

### 8.5 GEO-watcher 后台显示未配置平台

检查 `CRAWLER_PLATFORM_ENDPOINTS`：

- key 必须是平台 ID
- value 必须是接口路径或完整 URL
- 项目 B 推荐使用完整 URL

## 9. 最小配置清单

GEO-watcher `backend/.env` 示例：

```env
CRAWLER_BASE_URL=http://geo-watch-crawler:8000
CRAWLER_API_KEY=<GEO-watcher 调 crawler 的 key>
CRAWLER_CALLBACK_URL=http://geo-watcher-backend:3005/api/integrations/crawler/uploads
CRAWLER_CALLBACK_API_KEY=<双方约定的 callback key>
CRAWLER_SHARED_NETWORK=crawler-shared
CRAWLER_BACKEND_ALIAS=geo-watcher-backend
PROJECT_B_SHARED_NETWORK=project-b-crawler-shared
PROJECT_B_BACKEND_ALIAS=geo-watcher-backend
CRAWLER_PLATFORM_ENDPOINTS={"yuanbao":"/api/v1/tasks/yuanbao","project-b-platform":"http://project-b-crawler:8000/api/v1/tasks/project-b-platform"}
```

项目 B `.env` 示例：

```env
PROJECT_B_SHARED_NETWORK=project-b-crawler-shared
PROJECT_B_CRAWLER_ALIAS=project-b-crawler
CRAWLER_CALLBACK_URL=http://geo-watcher-backend:3005/api/integrations/crawler/uploads
CRAWLER_CALLBACK_API_KEY=<双方约定的 callback key>
SCRAPER_API_KEY=<GEO-watcher 调 crawler 的 key>
```

## 10. 最终原则

- 正式 Docker 环境不要用 `127.0.0.1` 互相访问
- `0.0.0.0` 只能用于监听，不能作为请求目标
- 项目 A 和项目 B 不要共用 crawler network
- 每个 crawler 项目单独一个 external network
- GEO-watcher backend 可以加入多个 crawler external network
- 新 crawler 项目的平台 endpoint 推荐使用完整 URL
- callback 字段必须原样使用 GEO-watcher 下发的 `env`
